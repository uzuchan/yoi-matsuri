# ART DIRECTION — 宵祭(よいまつり)

所有者: art-director / 最終更新: 2026-06-14 / 版: 1.1(Vertical Slice / T-009 雰囲気仕上げレビュー反映)

## 1. ビジュアルコンセプト

「提灯の灯りだけが暖かい、青い夏の夜」。低ポリゴンのプロシージャル造形を、暖色光源と濃い夜のフォグで包む。写実ではなく、記憶の中の祭りの絵。

- コントラスト軸: 寒色の夜(背景) × 暖色の灯り(提灯・屋台)。この対比を壊す中間色光源を置かない
- 外部画像アセット禁止。すべてThree.jsのジオメトリ+マテリアル+シェーダで表現

**群衆シルエットの明度持ち上げ例外(夜の前景での背景分離)**: 群衆シルエットの基準色は `#0d1126`。ただし夜の前景地面(土 `#3a3148` が暗く沈んだ領域)と明度が拮抗し、群衆が背景へ埋没して「人がいる」と読めなくなる場合に限り、フォグ色 `#141a38` を**上限**として明度のみを持ち上げ、背景から最小限分離してよい。このとき色相は寒色のまま据え置き、新規の色相を持ち込まない(`#141a38` は本パレットのフォグ色であり、新色の追加には当たらない)。基準色 `#0d1126` を超えて明るくする根拠は背景分離のみで、`#141a38` を超えてはならない(T-002 ループ2で適用)。

**金魚すくいシーンの器・背景・紙劣化の明確化(T-006 で確定)**: §2 表で水面・金魚・ポイの色は定義済みだが、(a)水槽の器(縁・壁・底)の色、(b)シーン背景、(c)紙の段階劣化の境界が未定義だった。以下に数値で固定する(いずれも §2 既存パレット内。新色相は持ち込まない)。

- **水槽の器(縁/壁/底)**: 縁は参道の土と同じ `#3a3148`(夜に沈んだ器として参道と共通の寒色)。鉢の側壁・底は `#3a3148` を 0.6 倍した暗色(縁より沈めて深さを示す)、水の底面は水面色 `#1e4d6b` を 0.4 倍した暗色。器は無発光(被ライティングのみ)で、夜祭りの寒色トーンを保つ。
- **水面の弱発光**: 水面 `#1e4d6b` op0.85 は、夜のフォグ背景に沈んで「黒い穴」に見えるのを防ぐため emissive `#1e4d6b` を emissiveIntensity 0.18 で弱く重ねてよい(器の中の水が淡く光る)。色相は水面色のまま。輝度は屋台前(§7-4)・金魚(主役)を超えてはならない。
- **シーン背景**: goldfish シーンは俯瞰のため空グラデーション(§2 夜空)は画面に入らない。背景はフォグ色 `#141a38` の単色とし、approach の夜のトーン(フォグ色は両シーン共通)と地続きに見せる。フォグは近接俯瞰スケールに合わせ `FogExp2(0x141a38, 0.12)` を許容する(§4 の参道用 0.028 はワールドスケール用。goldfish は水槽が camera から約1.5mの近接スケールのため密度を上げて器の縁を背景へ自然に馴染ませる。色は §4 と同一 `#141a38`)。
- **ポイ紙の段階劣化(GDD §4.6 破損予兆)**: 紙 `#f5f0e8` の不透明度は耐久比 r=durabilityRatio に対し基本 `op = 0.4 + 0.55·r`(r=1で0.95、r=0で0.4)。残30以下(r≤0.3)では破損予兆として透けをさらに強め、係数 `(0.55 + 0.45·(r/0.3))` を上記に乗じる(r=0.3で×1.0=境界連続、r=0で×0.55)。結果として r=0 付近では実効 op ≈ 0.22 まで透ける。HUD ゲージは r≤0.3 で警告色(金魚赤 `#e84a30`)+点滅に切り替える(§2 UIアクセント運用と別に、警告は金魚赤を流用してよい)。

**プレイヤーの視認性(T-003 で確定)**: プレイヤーは三人称追従カメラ(§5 approach)で常時画面に映る操作対象であり、群衆シルエットより**読みやすく**なければならない(GDD「三人称見下ろし気味の追従カメラ。プレイヤーは参道を歩く」)。問題: T-003 実装は胴を `MeshLambertMaterial`(被ライティング)で作ったため、入口や屋台横の影域では入射光が乏しく胴が純黒近くまで沈み、暖色帯のみが頼りになっていた(群衆=base色を持ち上げた**無発光**より暗く見える逆転)。
方針:
1. プレイヤー胴・頭は**寒色の固定床値 `#1b2240`(luma≈30)を保証する**。群衆の上限フォグ色 `#141a38`(luma≈27)より一段明るく、影域でもシルエットが読める。実装は無発光のbase色(`MeshBasicMaterial` 相当)か、`emissive #1b2240 / emissiveIntensity 1.0` を base に重ね、照明域では加えて明るくなる形のいずれか。**胴が照明の有無で純黒へ沈まないこと**が要件。
2. 暖色アクセント帯は祭りの法被/手ぬぐいの示唆として**残すが増やさない**。色 `#ff9d45`、emissiveIntensity ≤0.35、面積は胴の上部の細帯1本のみ。帯がプレイヤーの主たる視認手段になってはならない(§7-4 の屋台前=最暖・最明の視線誘導と競合させないため、暖色面積は屋台・提灯より常に小さく保つ)。
3. プレイヤーの暖色がシーンで一番明るい暖色になってはならない。最暖・最明は屋台前(§7-4)。プレイヤーの帯は「いる」と読める最小限で足りる。

## 3. 造形仕様(プロシージャル。単位: m)

- **提灯**: 球を上下に潰した楕円体(径0.35、高さ0.45)+上下に円筒の枠。参道両脇に2.5m間隔で各側24個、高さ2.6mのワイヤーに吊る。各提灯はわずかに揺れる(±2°、周期3〜5sでランダム位相)
- **鳥居**: 参道終端(z=-60付近)に高さ8m。シルエット重視、ディテール不要
- **金魚すくい屋台**: 間口3m×奥行2m。紅白幕、カウンター、屋根、裸電球2個(`#ffd166`)、水槽(楕円)。店主は単純化した人型シルエット+前掛け
- **群衆シルエット**: 単純化人型(高さ1.5〜1.8m)を参道脇に15〜20体。`#0d1126` の無発光マテリアル(夜の前景での背景分離は §2 の明度持ち上げ例外で上限フォグ色 `#141a38` まで許容)。ゆっくり揺れる(歩行はVertical Sliceでは不要)。**揺れの数値(T-009 で確定)**: 各個体は足元(y=0)を支点に前後へ ±3°(±数度)、周期4〜7s、index ベースの決定論的位相で正弦揺らす。提灯(±2°/周期3〜5s)よりわずかに大きく遅い「立っている人の重心移動」とし、静止に見せない(§7-3)。歩行ボブ・移動は付けない(立ち止まった群衆)
- **プレイヤー**: 単純化人型・身長 ~1.7m。胴=カプセル(半径0.22、円柱部1.0)+頭=球(半径0.17)、足元 y=0。胴・頭は寒色 `#1b2240`(§2 視認性方針: 照明の有無で純黒へ沈ませない床値を保証)。暖色アクセント帯=胴上部に細帯1本のみ(`#ff9d45`、emissiveIntensity ≤0.35、半径は胴+0.015、高さ0.12)。歩行アニメは本スライス範囲外(静的造形のみ)。新規色相は持ち込まない(全て §2 内)
- **金魚**: 紡錘形ボディ+三角尾びれ(計3メッシュ程度)。泳ぎに合わせ尾を±20°振る
- **花火**: パーティクル(1発120〜200粒、加算合成、重力落下+減衰、寿命1.8s)。打ち上げ筋→開花→残光の3段階。

  **花火の見栄え数値(T-009 で確定)**: 上記の段数・粒数・寿命に加え、夜空での発色と白飽和の扱いを以下に固定する(すべて §2 花火3色 `#ff6b9d`/`#ffd166`/`#4ecdc4` の範囲内。新色相は持ち込まない)。
  - **粒サイズ**: `PointsMaterial.size = 0.45`(sizeAttenuation。開花点 z≈-42〜-52m の遠方でも個々の火花が散って見え、近すぎて塊にならない)。
  - **加算合成の白飽和対策**: 開花の輝度ピークは `alpha ≤ 0.85`(1.0 にしない)。粒を小さく保つ(0.45)ことと合わせ、加算合成で粒が重なっても中心が純白に飛ばず、3色の色相が読めること。**判定基準**: 開花の塊が一様な白い球に見えてはならない。中心はやや明るくとも、外周の火花で `#ff6b9d`/`#ffd166`/`#4ecdc4` のいずれかの色相が識別できること(T-009 レビューでは中心がやや白飛び寄りで「色相がぎりぎり」と判定。VS は許容するが、出荷前に粒サイズ縮小 or ピーク輝度をさらに下げて色相の可読性を上げる余地あり=下記 T-011 申し送り)。
  - **残光フェード**: 寿命の前半45%は輝度ピーク維持、後半55%で線形に 0 へ。打ち上げ筋(launch)は burst 点の下へ約3mの細い尾とし、開花とつながった1発に見せる。
  - **開花点(視認性)**: approach カメラ(高さ3.2m・俯角15°・FOV55°)で画面上部の夜空に見えるよう、高さ10〜14m(鳥居 y=8 の上)・奥行 z≈-42〜-52m(鳥居の向こう)・x∈±10m に散らす。ライトは増やさない(加算合成の Points のみで夜空に映す。§6 の PointLight 6灯予算を消費しない)。
  - **打ち上げ間隔**: 30〜45s(GDD §2)。VS デモ視認性のため初回のみ早め(〜10s)。提灯・屋台の暖色が常時の主役で、花火は夜空のアクセント(常時連発しない)。

## 5. カメラ

- approach: プレイヤー後方5m・高さ3.2m・俯角15°の追従カメラ(追従lag 0.15s)。FOV 55°
- goldfish: 水槽真上やや手前から俯角70°固定。水槽が画面の70%を占める
- result: 屋台正面、店主が中央に入る固定カメラ

**result カメラの確定(T-007 で裁定 — 背景再利用の乖離を却下し専用カメラを要求)**: T-007 実装は背景に ApproachScene.render を再利用し(終了直前のプレイヤー追従カメラ位置で固定)、屋台が右端・店主が画面外という構図になっていた(reports/screenshots/T-007-result.png)。これは §5 result「屋台正面、店主が中央」と乖離し、かつ PRODUCT_VISION 体験目標3「小さな物語=店主の反応と報酬で『屋台で遊んだ』記憶を作る」の余韻を損なう(店主のセリフが出ているのに当の店主が画面にいない)。したがって **背景再利用は許容せず(a を却下)、result 専用の固定カメラを要求する(b を採択)**。世界(world)は ApproachScene 所有のまま再利用してよいが、描画に使うカメラのみ result 専用へ差し替える。専用カメラの数値仕様を以下に固定する(approach カメラとは別インスタンス。result シーン中のみ使用):

  - **注視点(lookAt)**: 屋台の店主の頭部付近。屋台中心を世界座標 S=(stall.x, stall.y, stall.z) とし、注視点は S を基準に y=+1.5m(店主の頭の高さ)。店主が画面横方向の中央(±5% 以内)に来ること。
  - **カメラ位置**: 屋台の正面(参道側=店主が客と対面する向き)から、屋台中心へ向かって水平距離 4.5m・高さ 1.8m。俯角は注視点(店主頭部 y=1.5)へ向けて約 4〜8°(ほぼ水平、店主とわずかに見上げる客の目線)。屋台の「正面」とは店主の前掛けが見える側(approach でプレイヤーが屋台に対面した向き)であり、屋台側面・背面から撮らない。
  - **FOV**: 50°(approach の 55° よりやや寄り、屋台+店主+裸電球2個が画面に収まり、店主が画面縦の中央〜やや上に来る)。
  - **構図要件**: 画面内に (1)店主シルエット+前掛けが中央、(2)屋台の紅白幕と裸電球2個(§4 屋台光が画面内で最も明るい=§7-4 を結果画面でも満たす)、(3)背景に提灯列が最低2個以上(寒色の夜×暖色の灯りの対比 §7-1 を保つ)。鳥居は任意。結果パネル(画面中央の80%透過)が店主の上半身に被るのは可だが、店主の頭〜肩が左右どちらかにはみ出して読めること。
  - **固定**: result 中はカメラを動かさない(追従・揺れなし)。プレイヤー造形は画面に入っても入らなくてもよい(主役は店主)。

  担当への差し戻し(4点セット): 対象オブジェクト=result シーンの背景描画カメラ(src/scenes/result/ResultScene.ts が ApproachScene.render を再利用している箇所)/ 現状値=approach 追従カメラ(プレイヤー後方5m・高さ3.2m・俯角15°・FOV55°、終了直前位置で固定。屋台が右端・店主が画面外)/ 期待値=上記 result 専用固定カメラ(屋台正面・水平距離4.5m・高さ1.8m・俯角約4〜8°・FOV50°、店主頭部 y=1.5 を注視・店主が画面横中央±5%)/ 該当仕様=ART §5 result・§7-1・§7-4、PRODUCT_VISION 体験目標3。実装はワールド再利用のまま専用 Camera を1つ追加し render で差し替える方針でよい(world 二重生成は不要 / 性能予算 §6 を増やさない)。

**result カメラの「正面」の確定(T-007 ループ2 で裁定 — 前掛け向きの矛盾を解消)**: 上の差し戻しを gameplay-engineer が解消し(ResultScene に result 専用固定カメラを追加し renderWith で描画)、reports/screenshots/T-007-result.png で屋台が画面中央へ来た(屋台が右端・店主が画面外という却下理由は解消)。実装報告で「ART §5 は『正面=前掛けが見える側』と記すが、既存 world 造形(src/world/stall.ts、group rotation.y=-90°)では前掛け(apron)・紅白幕の正面が参道と逆向き(world +x 側)を向く一方、プレイヤーが屋台に対面した側・屋台 PointLight(§7-4 屋台前を最も明るく)はいずれも world -x 側にある。両立しないため -x 側カメラを採用した」との確認があった。座標を裏取りしたところ事実と一致する(紅白幕の正面 world x≈6.3、apron 正面は +x 向き / 店主 world≈(4.5,-,-25.7)、裸電球 world≈(5,2.1,-26.9)・(5,2.1,-25.1)、屋台 PointLight world≈(4,-,-26))。

裁定: **-x 側カメラを許容する(前掛け向きの是正は要求しない)。** §5 の「正面=前掛けが見える側」の語は撤回し、以後 result の「正面」は (1)approach でプレイヤーが屋台に対面した側=-x 側、(2)§7-4 屋台前(裸電球・屋台 PointLight)が画面内で最も明るくなる側、の2点で定義する。前掛けの面を是正してカメラ側へ向ける利得は小さい — 店主は §3 通り無発光の暗色シルエット(`#0d1126` / MeshBasicMaterial)であり、前掛けの面はどの角度からも「前掛け」として読めない。一方 +x 側へ撮ると §7-4(最明=屋台前)と approach 継続性を同時に失う。よって VS 品質としては「店主シルエットが画面中央付近に見える」ことで十分とし、前掛けの面の可読性は要件にしない。environment-engineer への是正依頼は出さない。

T-007 ループ2 残課題(Minor / result カメラ解消の妨げにはしない): 店主シルエット(near-black `#0d1126`)が中央の 80% 透過パネル背後に隠れ、かつ夜の暗背景と明度が拮抗して「頭〜肩が左右へはみ出して読める」(§5 構図要件3)が現状成立していない。店主は意図的な暗色シルエットで余韻は本文テキスト(店主ラベル+セリフ)が担うため再差し戻し(不合格)はしないが、フォローとして次のいずれかを推奨する: (a)result 専用カメラの注視点を屋台中心ではなく店主頭部 world≈(4.5,1.66,-25.7)へ寄せ、店主をパネルの片側へ約 5〜10% ずらして頭〜肩を露出させる、または (b)store keeper シルエットに §2 群衆の明度持ち上げ例外と同等の背景分離(上限フォグ色 `#141a38`)を適用し、暗背景から最小限分離する。担当=gameplay-engineer((a)は ResultScene のカメラ注視点)/ environment-engineer((b)は src/world/stall.ts の keeperMaterial)。VS 範囲では任意。

**T-007 ループ2 残課題の解消(T-009 で確認 — 推奨案 a を採用)**: gameplay-engineer が推奨案(a)を実装した。result 専用カメラの注視点を店主頭部 X(4.5)・高さ1.5へ寄せ、カメラ rig 全体を world +z へ `RESULT_KEEPER_PAN_Z = 1.8m` パンして店主を画面中央から左へ約23%(画面 x≈23%)ずらした(中央パネル左端 x≈28% の左外に頭〜肩が出る)。パン後も裸電球2個・屋台前(§7-4 最明)は画面内に残り、approach 継続性も維持される。reports/screenshots/T-007-result.png(2026-06-14 撮影、ResultScene の T-009 改修後)で確認: 屋台・紅白幕・裸電球2個が画面左に最明として残り(§7-4 ✓)、結果パネルが中央に被っても店主側が左外へ寄り、画面右下にはプレイヤー造形(寒色胴+暖色帯)が入る構図になっている。**店主シルエット自体は依然 near-black `#0d1126`(被ライティング無しのベース色)で暗背景に対し可読性が低い**(店主の前掛け apron は MeshLambertMaterial 白だが world +x 側を向くため -x カメラからは見えない)。VS としては「店主側が中央パネルの外へずれ、店主の位置が画面内に確保された(余韻は本文テキストが担う)」ことで §5 構図要件を満たすと判定する。前掛けの面の可読性・店主シルエットの明度持ち上げ(推奨案 b)は要件にしない(T-007 ループ2 裁定どおり)。Minor はクローズ。

## 6. 性能との取り決め(TECHNICAL_ARCHITECTUREと連動)

- 動的PointLightは合計6灯以下。ジオメトリはシーン全体で50k三角形以下
- 提灯・群衆はInstancedMesh必須
- この予算内で50 FPSを維持できない演出は削る(美しさより滑らかさ)
