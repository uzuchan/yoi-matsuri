## 2. 参道散策(approach)

- 三人称見下ろし気味の追従カメラ。プレイヤーは参道(幅8m、奥行き60m)を歩く
- 参道の両脇に提灯列と群衆シルエット。奥に鳥居。中腹の右側に金魚すくい屋台
- 移動: WASD/矢印キー(詳細はINTERACTION_SPEC)。走り無し。移動速度 `walkSpeed = 3.0 m/s`
- 屋台から半径 `interactRadius = 3.0 m` 以内でプロンプト「E: 屋台をのぞく」を表示
- 30〜45秒間隔で花火が打ち上がる(視覚+音。ゲームプレイへの影響なし)

### 3.2 結果別の店主の反応(resultシーンで表示)

| 結果(段) | 条件 | 店主のセリフ | 報酬 |
|---|---|---|---|
| 失敗 | 捕獲0匹 | 「ありゃー、破れちまったか。まあ祭りの夜は長いんだ、また挑戦しな!」 | 残念賞: ラムネ風アメ(`reward:candy`) |
| 成功 | 捕獲1〜2匹 | 「おっ、やるねえ!ほら、大事にしてやんなよ。」 | 金魚袋(`reward:bag-small`、捕獲した金魚入り) |
| 大成功 | 捕獲3匹以上 | 「うおっ、名人かい!?こりゃあ参った、特別だ!」 | 大きな金魚袋+出目金(`reward:bag-deluxe`) |

**段の判定はsecured数のみ**: 上表の「段(失敗/成功/大成功)」・店主セリフ・報酬は、終了理由(reason)に依らず確保数(secured)のみで決まる。境界は **0匹=失敗 / 1〜2匹=成功 / 3匹以上=大成功**(2匹=成功、3匹=大成功)。

**「捕獲」の定義(報酬段の判定基準)**: 上表の「捕獲n匹」は **お椀へ確保した数(secured)** を指す。ポイへ載せた(caught)が確保前にポイ破損・時間切れで失われた金魚は数えない。報酬段の判定値は結果画面に渡される `finished.caught`(= `secured`)であり、HUDの「確保数(お椀の金魚)」と一致する(§4.4 / §5)。

**見出し(結果画面のタイトル)は終了理由で分岐する**: 段が同じでも、見出し文言は終了理由(`finished.reason` = `'torn'`(破損)/ `'timeout'`(時間切れ)/ `'quit'`(途中退出。Esc))で次のとおり出し分ける。「破れていないのに『破れた』と出る」齟齬を防ぐためで、段判定(上表)とは独立した表示ルールである。

| 段 | reason | 見出し文言 |
|---|---|---|
| 失敗(0匹) | `torn`(破損) | `ポイが破れてしまった…` |
| 失敗(0匹) | `timeout`(時間切れ)/ `quit`(途中退出) | `すくえなかった…` |
| 成功(1〜2匹) | 任意(reason非依存) | `金魚をすくった!` |
| 大成功(3匹以上) | 任意(reason非依存) | `大漁!名人級!` |

- 成功・大成功段は1匹以上を確保しており、破損で終わっても「すくえた」事実は成立するため、見出しは従来どおりreason非依存(`金魚をすくった!` / `大漁!名人級!`)とする。reason分岐は失敗段(0匹)に限る。
- 失敗段の店主セリフは現状reason非依存(`ありゃー、破れちまったか。…`)だが、これは破損を前提にした文言である。`timeout`/`quit`(=破れていない)の0匹時は見出し(`すくえなかった…`)と齟齬するため、失敗段のセリフも見出しに合わせて出し分ける。

| 段 | reason | 店主セリフ |
|---|---|---|
| 失敗(0匹) | `torn`(破損) | 「ありゃー、破れちまったか。まあ祭りの夜は長いんだ、また挑戦しな!」 |
| 失敗(0匹) | `timeout`(時間切れ)/ `quit`(途中退出) | 「おっと時間だ。今日はご縁がなかったか。…また挑戦しな!」 |

- 見出し `ポイが破れてしまった…` / `金魚をすくった!` / `大漁!名人級!` の3文言は INTERACTION_SPEC §4 に既定。新設の失敗見出し `すくえなかった…` は本GDDで初定義のため、interaction-designer に INTERACTION_SPEC §4 への追記を依頼する(本タスクの報告で連絡)。

報酬は結果画面で視覚表示し、HUDの所持品スロットに反映する(Vertical Sliceでは表示のみで使用機能は無し。ただし表示は実動作であること)。

## 4. 金魚すくい(goldfish)

### 4.1 構成

俯瞰視点の水槽(画面中央、楕円形 直径約1.2m相当)。金魚8匹が遊泳。画面端にお椀。
プレイヤーはポイ(直径 `poiRadius = 0.09 m` の円形すくい網)をマウスで操作する。

### 4.2 操作モデル

1. **追従**: ポイはカーソル位置へ慣性付きで追従する(即時追従ではない。後述の物理)
2. **沈める**: マウス押下でポイが水中へ沈む(`dipDepth = 0.04 m`)。水中では水の抵抗が効く
3. **持ち上げ**: マウス解放でポイが水面上へ戻る。このとき円内に金魚がいれば捕獲判定
4. **運ぶ**: 捕獲中の金魚はポイに乗る。お椀の上で再度沈める動作(クリック)で確保確定
5. キーボード代替: 矢印キーでポイ移動、Spaceで沈める/持ち上げ(INTERACTION_SPEC参照)

### 4.3 物理パラメータ(初期値。実装は src/game/goldfish/params.ts に一元化)

| 変数名 | 初期値 | 単位 | 意味 |
|---|---|---|---|
| `poiFollowLag` | 0.12 | s | カーソル追従の時定数(空中) |
| `waterDragFactor` | 3.5 | 倍 | 水中での追従時定数の倍率(水の抵抗。水中lag = 0.12×3.5) |
| `paperDurability` | 100 | pt | 紙の耐久値(満タン) |
| `wetDamagePerSec` | 4.0 | pt/s | 水中にいるだけで受ける毎秒ダメージ |
| `speedDamageCoeff` | 220 | pt·s²/m² | 水中移動ダメージ係数(damage/s = coeff × speed²) |
| `fishWeightDamage` | 12 | pt | 金魚を載せて持ち上げた瞬間の固定ダメージ |
| `liftSpeedMax` | 0.35 | m/s | 持ち上げ時にこれより速いポイ移動だと金魚がこぼれる |
| `fishEscapeRadius` | 0.18 | m | 金魚がポイから逃げ始める距離(水中のポイのみ) |
| `fishCruiseSpeed` | 0.10 | m/s | 金魚の通常遊泳速度 |
| `fishFleeSpeed` | 0.45 | m/s | 金魚の逃避速度 |
| `sessionTimeLimit` | 60 | s | 制限時間 |
| `fishCount` | 8 | 匹 | 水槽内の金魚数 |

### 4.4 ルール

- 耐久値が0になるとポイが破れ、その場でセッション終了(載っていた金魚は水槽へ戻る)
- **結果の捕獲数 = お椀へ確保した数(secured)**。ポイへ載せただけ(caught/onPoi)で確保前に破損・時間切れになった金魚は結果に数えない。報酬段(§3.2)も `finished.caught = secured` を基準とする。これは「お椀に入れて初めて自分の金魚」という金魚すくいの実体験に沿う(載せた瞬間ではなく運び切る緊張までを1捕獲とする設計)
- 時間切れでもセッション終了。終了時、確保数 > 0 なら status=won
- **複数捕獲は可**: 1回の持ち上げで、ポイ円(`poiRadius`)内にいる遊泳中の金魚を**すべて同時に捕獲する**。捕獲ごとに金魚荷重ダメージ `fishWeightDamage`(12pt)が**1匹につき1回**加算される(2匹同時=24pt、3匹同時=36pt)。狙って群れを一網打ちにする「上手さの報酬」を許容しつつ、同時捕獲は耐久を比例して激しく消耗させ、運搬中の破損リスクを高めることで大成功(3匹以上=確保3匹以上)を割に合いにくくする(下記の設計意図を参照)
- 耐久値はHUDにポイの紙の見た目劣化(透け・ヨレ)+ゲージで表示する
- 「速く動かすと一気に破れる / そっと動かせば長持ちする」が体感できること(speed²項が支配的)
- 設計意図: 1回目のプレイで平均1〜2匹(確保ベース)、3匹以上は丁寧な操作の習熟が必要なバランス。複数捕獲を許容しても、(1) ポイ円(半径0.09m)は狭く逃避半径(0.18m)より小さいため複数同時に円内へ収めるのが難しい、(2) 同時捕獲ぶんの荷重ダメージ(12pt×匹数)で運搬余力が削られる、(3) 確保は1匹ずつでなくポイ上の全匹を一度に確保するが、運搬中に破れれば全て失う、の3点が「3匹以上は習熟」を担保する。よって複数捕獲を許可しても平均1〜2匹のバランスは崩れない見込み(実体感はT-006プレイ後に再評価する)

### 4.5 金魚AI

- 通常: ランダム目標点へのwander(速度 `fishCruiseSpeed`)。水槽境界で滑らかに転回
- 逃避: 水中のポイが `fishEscapeRadius` 内に来ると、ポイと逆方向へ `fishFleeSpeed` で逃げる(0.8s持続)
- 捕獲判定: ポイ持ち上げ(沈める→持ち上げの立ち上がりエッジ)時、金魚の中心がポイ円(`poiRadius`)内 かつ ポイ水平速度 ≤ `liftSpeedMax` で捕獲。速度超過時は円内の金魚すべてが跳ねて水槽へ戻る(失敗演出+耐久ダメージは無し)
- **複数同時捕獲**: 1回の持ち上げで条件を満たす金魚が複数いれば、その全匹を同時に捕獲する(§4.4)。速度ゲートは1回の判定で円内全匹に一括適用される(全捕獲 or 全失敗。一部だけ捕獲はしない)。捕獲した各匹に荷重ダメージ `fishWeightDamage` を個別加算する

### 4.6 手触りの調整指針(ダメージ感・捕獲緊張・破損予兆)

T-005 のロジック/パラメータがこの手触りを満たすことを確認済み(数値根拠つき)。T-006(描画)実装時はこの体感が崩れないよう以下を基準とする。

**ダメージ感(そっと=長持ち / 速く=即破れる)**: 水中ダメージは `wetDamagePerSec + speedDamageCoeff × speed²` [pt/s]。満タン100ptを基準に、

| 水中ポイ速度 | 毎秒ダメージ | 満タン(100pt)消尽まで | 体感 |
|---|---|---|---|
| 静止(0 m/s) | 4.0 pt/s | 25 s | 沈めて待てる猶予はあるが永遠ではない |
| 0.1 m/s(=金魚の遊泳速度) | 6.2 pt/s | 約16 s | そっと寄せれば十分長持ち |
| 0.35 m/s(=liftSpeedMax) | 約31 pt/s | 約3.2 s | 雑に動かすと一気に減る |
| 1.0 m/s | 224 pt/s | 約0.45 s | ほぼ即破れ(速い操作への明確な罰) |

speed²項が支配的で、「そっと=長持ち / 速く=即破れる」が成立する(✓ §4.4 の意図に乗っている)。突然死ではなく、速度を上げた本人の操作に破損理由が帰属する。

**捕獲緊張(平均1〜2匹のバランス)**: 逃避半径(`fishEscapeRadius` 0.18m)> 捕獲円(`poiRadius` 0.09m)のため、水中ポイを近づけると捕獲できる前に金魚が `fishFleeSpeed`(0.45 m/s)で逃げ始める。これを成功させるには、(1) 金魚が逃げる前に素早く沈めて持ち上げる、(2) ただし持ち上げ速度は `liftSpeedMax`(0.35 m/s)以下に抑える、という相反する要求を両立する必要がある。この「速く、でも丁寧に」の間合いが習熟要素を生み、平均1〜2匹・3匹以上は習熟、というバランスの数値的裏付けとなる(実体感はT-006後に再評価)。

**破損予兆(突然死に感じさせない)**: 残耐久が `PAPER_WARNING_THRESHOLD`(30pt)以下に初めて到達した時点で `paper-warning` を1回だけ通知し、HUDのゲージ+紙の見た目劣化(透け・ヨレ)と警告音で「もう危ない」を伝える(§5 / AUDIO_SPEC §4)。30ptは静止でも約7.5秒、0.1m/sなら約4.8秒の猶予に相当し、警告から破損まで操作を緩める余地がある。**T-006への要請**: 残30以下では紙の見た目劣化を段階的に強める(残0へ向け透け・ヨレが進行)ことで、ゲージを見ていなくても破損が近いと分かるようにする。これにより「突然破れた」と感じさせない(§4.4 / §6)。

## 6. バランス調整の指針

- 調整はこのGDDのパラメータ表を更新 → gameplay-engineerが params.ts に反映、の順。コード直接変更によるバランス調整は禁止
- 「破れる理由がプレイヤーに伝わるか」を最優先(突然死に感じさせない: 耐久ゲージと紙の見た目劣化、破れる直前の警告音)。具体的な手触り基準と数値は §4.6 を参照
- 再評価ポイント(T-006プレイ後): 複数同時捕獲で大成功(確保3匹以上)が容易になりすぎていないか。もし容易すぎる場合の調整候補は、(1) `fishWeightDamage` を上げて同時捕獲の運搬リスクを増す、(2) `poiRadius` を狭める、(3) `liftSpeedMax` を下げる、のいずれか。いずれもこのGDDの数値を更新してからgameplay-engineerへ依頼する

### 6.1 再評価の結果(T-011 出荷判定時の所見 / 2026-06-14)

T-011 で、確定済みの `GoldfishSession`(実プレイと同じ物理・AI・ダメージ)を用い、人間の操作を模した方策(カーソルジッタ付きで最寄りの金魚へ寄せ→短く沈めて低速で持ち上げ→お椀へ運んで確保、を反復)で 200〜300 シードを試走した。結果、**§4.4 の「初回プレイ平均1〜2匹/3匹以上は習熟」という見込みは成立していない**:

- 初級相当(ジッタ 0.06m・沈め保持 8 フレーム)でも 300/300 が大成功(確保3匹以上)、確保数の平均 ≈ 5.2 匹。
- 中級相当(ジッタ 0.03m)で平均 ≈ 7.2 匹、熟練相当(ジッタ 0.012m)で全試行 8 匹完取。

原因(§4.4 が「3匹以上は習熟」を担保するとした 3 点が、現実的な方策でいずれも無効化される):

1. **逃避が発動しない**: 金魚は「水中のポイ」が `fishEscapeRadius` 内に入ったときのみ逃げる(§4.5)。ポイを**空中**で金魚直上へ寄せてから一瞬だけ沈めて持ち上げると、逃避が立ち上がる前に捕獲が決まる。`poiRadius`(0.09m)< `fishEscapeRadius`(0.18m)という関門は、空中接近の前では機能しない。
2. **運搬で耐久が減らない**: 耐久ダメージは**水中のみ**で発生する(§4.3 / poi.ts)。確保のための運搬は空中で行えるため無ダメージ。`fishWeightDamage`(12pt)も、1匹ずつ短時間で確保していけば 100pt の予算に対して軽微で、運搬余力を削る抑止にならない。
3. **時間が余る**: 上記方策では破損は終盤にしか起きず(試走では平均で残り時間 ≈ 54s/60s の時点で既に大量確保)、`sessionTimeLimit` も制約にならない。

→ **判定への影響**: ポイ速度・水抵抗・紙耐久が結果に効くこと自体(VS要件5 / 体験目標2 の手触りの核)は params-influence テストと上記試走の通り成立しており、これは満たす。一方で「そっと寄せて一喜一憂する」ねらいの**難度バランスが緩く、大成功が常態化する**。出荷の可否を分ける Critical ではない(物語・報酬は破綻せず、むしろ大成功演出が高頻度で出る)が、体験目標2の「緊張感」を弱めるため、出荷後の最優先のバランス調整対象とする。

**採るべき調整方針(次のバランスタスクで GDD 数値を更新してから gameplay-engineer へ依頼)**: 単一パラメータの微調整では上記の構造的抜け道(空中接近・空中運搬)を塞げない。**仕様レベルの追加**が要る。優先順:

- **(A) 確保にも水中接触を要求する/運搬中も微少な水中ダメージを与える等、「空中で完結させられない」制約を入れる**(逃避と耐久の両方を実際に通らせる)。最も効果が大きいが §4.2/§4.4 の操作・ルール変更を伴うため、interaction-designer と協議のうえ別タスクで仕様確定する。
- **(B) `fishEscapeRadius` を維持しつつ、ポイが「空中で金魚直上に静止していると金魚が散る(警戒 wander)」挙動を金魚AIへ追加**し、空中での事前ロックを難しくする。
- **(C) 暫定の数値調整(構造は変えず難度だけ上げる)**: `liftSpeedMax` を 0.35→0.20 程度へ下げ(丁寧な持ち上げをより要求)、`poiRadius` を 0.09→0.07 へ狭める(同時捕獲と命中を厳しく)、`fishWeightDamage` を 12→18 へ上げる。効果は限定的だが (A)/(B) 着手前の緩和として有効。

いずれも本 GDD の数値・仕様を更新してから gameplay-engineer(仕様追加が要る場合は interaction-designer も)へ依頼する。**この調整は出荷判定をブロックしない**(下記 T-011 判定参照)。

### 8.2 お面屋(原型E CHOICE / 屋台#19)

#### ゲームルール

- 物理なし。選択式。壁に並ぶお面(5種)をカーソル/キーボードで選び、Enterで確定するだけ
- **失敗概念なし**。Esc退出のみ score=0。「お面を選ぶ行為そのもの」が体験で、正誤を問わない

#### 報酬の扱いについて(設計方針の明示)

現フレームワーク(resolveStallResult)は **tier 解決で1種の報酬 ID を返す**。このため「選んだお面そのもの(キツネ/ひょっとこ/おかめ/アニメ)が個別に報酬として渡る」ことは現仕様の範囲外であり、成功段は汎用 `reward:mask:chosen`(「えらんだお面」) が報酬となる。大成功(般若)のみ固有 ID(`reward:mask:hannya`)を返す。

この仕様は現 Vertical Slice の範囲では許容する。コレクション図鑑機能(MINIGAME_ARCHETYPES §2 #19)を実装する際に、`metrics.chosenIndex` から選んだお面の固有 ID を取得する拡張を行う(gameplay-engineer への依頼事項。その時点でこの段落を更新する)。
